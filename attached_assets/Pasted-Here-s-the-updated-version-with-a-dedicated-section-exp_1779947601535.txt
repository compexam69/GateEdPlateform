🟢 Checkpoint 7: Completely Remove Resend Integration & Replace with Supabase-Only Email Architecture

Goal: Completely remove all Resend-related functionality, dependencies, and configurations from the existing codebase and replace the current email architecture with a Supabase-only solution that works properly without a verified custom domain and remains fully compatible with free-tier development/testing environments.

Context: The project currently has:

* Supabase authentication with built-in email verification
* Partial or existing Resend integration for custom transactional emails

Current issue:

* No verified custom domain exists.
* The project is running on free-tier infrastructure.
* Resend API key exists, but domain verification is NOT configured.

Existing codebase analysis shows:

| Scenario                           | Auth (verify/reset) | Welcome email    | Approval email   | Storage alert    |
| ---------------------------------- | ------------------- | ---------------- | ---------------- | ---------------- |
| No domain, no Resend setup         | ✅ Works (Supabase)  | ❌ Fails silently | ❌ Fails silently | ❌ Fails silently |
| Resend key set, no verified domain | ✅ Works (Supabase)  | ❌ Resend rejects | ❌ Resend rejects | ❌ Resend rejects |

Problem:

* Supabase authentication emails already work correctly.
* Resend-based emails fail because the domain is not verified.
* The existing codebase may contain:

  * Broken Resend integrations
  * Silent email failures
  * Dead serverless functions
  * Unused packages
  * Redundant email logic
  * Environment variables that are no longer useful

New Requirement:

* Completely remove ALL Resend integrations from the project.
* Replace all possible email functionality with Supabase-only implementations wherever technically possible.
* Ensure the project remains fully functional without:

  * Verified domain
  * Resend service
  * External SMTP provider
* Ensure the project remains testing-ready and free-tier friendly.

Instructions:

1. Analyze Entire Existing Email Architecture:

* Review:

  * Authentication/signup flow
  * Email verification flow
  * Password reset flow
  * Welcome email flow
  * Approval email flow
  * Storage alert email flow
  * Serverless/email utility functions
  * Environment variables
  * API integrations
  * Installed packages/dependencies
  * Supabase auth configuration
* Identify:

  * Every place where Resend is used
  * Every Resend API call
  * Every Resend-related package/import
  * Any duplicate email logic
  * Any silent email failure points

2. Completely Remove Resend Integration:

* Remove:

  * Resend SDK/packages
  * Resend API calls
  * sendWelcomeEmail()
  * sendApprovalEmail()
  * sendStorageAlertEmail()
  * Resend environment variables
  * Unused email helper utilities
  * Dead serverless/email functions
  * Broken fallback logic
* Ensure:

  * No Resend dependency remains anywhere in the codebase
  * No hidden imports/configurations remain
  * No build/runtime errors occur after removal

3. Replace Email Functionality with Supabase Where Possible:

* Use Supabase built-in functionality for:

  * Signup verification emails
  * Resend verification emails
  * Password reset emails
* Analyze whether additional custom flows can be replaced safely by:

  * In-app notifications
  * Dashboard alerts
  * Database-driven notifications
  * System banners/messages
    instead of external email providers.

4. Handle Unsupported Email Features Properly:

  * Identify which features CANNOT be replaced directly.
* For unsupported features such as:

  * “Welcome — account pending approval”

  * In-app notification cards
  * Dashboard status alerts
  * Approval banners/messages
  * Admin notification panels
  * Database notification logs
* Ensure:

  * UX remains understandable
  * No broken email expectations remain

  * Signup flow

  * Admin approval workflow

  * Route protection
  * Session handling

6. Remove Silent Failures:
