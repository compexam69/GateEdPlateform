🟢 Checkpoint X: Subject-Level Access Control System for Content Editor (Admin & Super Admin Dashboard)

Goal:

Implement a secure and production-ready subject-level access control system inside the Content Editor module for both Admin and Super Admin Dashboards, allowing controlled visibility and ownership of subjects while enforcing strict Super Admin isolation and preserving all existing content hierarchy workflows.

Context:

Currently, the Content Editor allows Admins and Super Admins to create/manage:

* Subjects
* Chapters
* Topics
* Lectures
* Exams/Tests
* Notes/PDFs

However, there is no dedicated subject-level role-based visibility/access control system.

As a result:

* Subjects may become globally visible unintentionally.
* Subject ownership isolation is not enforced properly.
* One Super Admin may gain access to another Super Admin’s subjects/content hierarchy automatically.
* Downstream content inheritance behavior is undefined.
* Existing content hierarchy permissions are inconsistent.

The platform now requires a centralized subject-level access control architecture that integrates with all previously implemented lecture/content/exam access-control systems.

New Requirement:

Add role-based multi-selection access control directly during Subject creation/editing inside the Content Editor for both:

* Super Admin Dashboard
* Admin Dashboard

The subject creator must be able to define who can access/manage/view that subject hierarchy.

Supported visibility targets:

* Creator Only
* Admins
* Students
* Explicit Super Admin sharing

Critical Business Rule:

* A Super Admin can access:

  * Their own subjects
  * Admin-created subjects
  * Student-visible subjects (if allowed)
* BUT:

  * One Super Admin MUST NOT automatically access another Super Admin’s subjects/content hierarchy.
* Cross-Super-Admin access requires:

  * Explicit sharing permission.

VERY IMPORTANT:

Subject-level permissions must properly integrate with:

* Chapters
* Topics
* Lectures
* Exams/Tests
* Notes/PDFs
* Analytics
* Search/filter systems

Permission inheritance and override behavior must be clearly defined and implemented safely.
