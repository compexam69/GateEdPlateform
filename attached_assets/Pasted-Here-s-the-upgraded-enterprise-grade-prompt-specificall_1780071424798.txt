🟢 Checkpoint X: Content Editor Lecture Access Control System (Role-Based Visibility + Super Admin Isolation)

Goal:

Implement a secure and production-ready role-based access control system inside the Content Editor module for lecture/content creation, ensuring creators can control who can access specific lecture content while enforcing strict Super Admin isolation and preserving all existing content workflows.

Context:

Currently, the Content Editor module allows Admins/Super Admins to create lecture content, but there is no flexible role-based visibility/access control system attached to lecture-level content.

As a result:

* Content visibility is overly broad or system-dependent.
* There is no granular access configuration.
* Super Admin content isolation is not enforced properly.
* One Super Admin may unintentionally gain visibility into another Super Admin’s lecture content.
* Existing lecture delivery workflows do not support role-based visibility filtering.

New Requirement:

Add a role-based multi-selection access control system directly inside the Content Editor lecture creation/edit workflow.

The creator must be able to define who can access a lecture/content item.

Supported access targets:

* Creator Only
* Admins
* Students
* Specific Super Admin access (explicit sharing only)

Critical Business Rule:

* Super Admins can view:

  * Their own lecture content
  * Admin-created lecture content
  * Student-accessible lecture content (if allowed)
* BUT:

  * One Super Admin MUST NOT automatically access another Super Admin’s lecture content.
* Cross-Super-Admin visibility requires:

  * Explicit permission/share access.

Examples:

* Creator only lecture
* Admin-only lecture
* Student-accessible lecture
* Shared lecture between selected Super Admins
* Mixed Admin + Student lecture access

* Whether lecture visibility fields already exist

* Potential permission/security loopholes

3. Implement Super Admin Isolation Logic

Critical Requirement:

One Super Admin MUST NOT automatically gain access to another Super Admin’s lecture content.

Implement:

* Explicit Super Admin sharing system.

Possible implementation approaches:

A. Shared Access Table (Recommended)

Create a mapping table:

lecture_shared_access

Fields:

* lecture_id
* owner_super_admin_id
* shared_super_admin_id
* created_at

OR

B. JSON-based shared access array

Only if architecture already supports JSON permissions cleanly.

Recommended:

Use relational mapping table for scalability/security.

Ensure:

* Isolation enforced at:

  * UI level
  * API level
  * Database/RLS level
* Unauthorized Super Admins cannot:

  * View
  * Search
  * Open
  * Query
  * Access APIs
    for restricted lecture content.
